# Explorar 2.0 — design (chat estilo Perplexity + Busca)

**Data:** 2026-07-07 · **Status:** aprovado (brainstorming) · **Origem:** feedback de UX após o chat RAG entrar em produção.

## Contexto

O épico de streaming (SSE) do "Explorar" foi concluído: o chat recupera fontes e gera a resposta token a token, ancorada e citada. Rodando de verdade, surgiram problemas de UX e uma dúvida de arquitetura de informação (IA):

1. Os **blocos de fontes não são interativos** — não dá pra abrir/pré-visualizar a fonte. O dado já chega (`documentId`, `kind`, `score`, `snippet`, `title`, `slug`); falta transformá-lo em algo visual e navegável.
2. O **cursor de streaming** fica solto numa linha abaixo do texto, sem integração com o chunk que está chegando.
3. **"Explorar" virou um chat**, mas o nome sugere "encontrar coisas". Encontrar-um-documento e responder-uma-pergunta são intenções diferentes.

## Decisões travadas

- **Separar chat de busca.** "Explorar" = **assistente/chat** (RAG). Criar uma **Busca** semântica separada (encontra trecho → abre o doc). Os dois backends já existem (`/explore` e `/search`).
- **Inspiração: Perplexity** para a UX do chat (citações inline + hover cards + barra de ações).
- **Fontes por hover** (não clique), com **ponte de hover**.
- Fatiar em 3 entregas independentes (1 milestone cada).

## Fatia 1 — UX do chat (AGORA)

Milestone: **M1 — Explorar: UX do chat**. Só front-end (`apps/web`). Sem mudança de backend.

### 1a. Fontes interativas — citações inline + hover card (Perplexity)

**Hoje:** bloco "Fontes" (lista de cards) no topo da resposta; `[1]` como texto solto.

**Alvo:**
- **Citações inline:** os marcadores `[n]` no texto viram **chips** clicáveis/hoveráveis. O mapeamento já existe: o LLM numera na mesma ordem de `sources`, então `[1]` → `sources[0]`.
- **Linha de fontes compacta** abaixo da resposta: pílulas pequenas (ícone doc/arquivo + título) + contador **"N fontes"**.
- **Hover card** (no chip ou na pílula):
  - Abre no **hover** (mouseenter), não no clique.
  - **Ponte de hover:** ao mover o cursor do gatilho para dentro do card, ele **não fecha**; só fecha quando o cursor sai **de ambos** (gatilho e card). Implementar com um pequeno *delay* de fechamento (~120ms) + a área do card contando como "dentro".
  - Conteúdo: título, trecho no contexto (mais linhas que o snippet atual, se disponível), e botão **"Abrir documento/arquivo"**.
- **Abrir:** `kind: 'native'` → editor (`/app/editor?id=`) ou página pública (`/d/:ownerId/:slug`); `kind: 'file'` → abre/baixa o arquivo. *(Rolar até o trecho exato = stretch, não obrigatório na v1.)*

**Decisão pendente (default proposto):** trocar o bloco de cards do topo por **citações inline + linha compacta** (mais Perplexity). Alternativa: manter o bloco e só torná-lo hoverável. → **Default: citações inline + linha compacta.**

**Arquivos:** `explore-view.tsx` (`MessageBubble`, novo componente `SourceHoverCard`/`CitationChip`), possível util de hover-intent.

**Critérios de aceite:**
- Hover num chip/pílula abre o card; mover pro card mantém aberto; sair dos dois fecha.
- Card mostra a fonte e leva ao documento/arquivo certo.
- Acessível por teclado (focus abre/fecha; `Esc` fecha) e não quebra em toque (fallback: clique abre).

**Emenda (2026-07-07, pós-teste em ambiente):** a linha de fontes ficou grande e repetida quando o RAG traz vários trechos do **mesmo documento** (top-K chunks). Ajustes:

- **Deduplicar por `documentId`** na linha compacta: N chunks do mesmo doc → **1 pílula** (mantém o de maior score para título/kind/link).
- **Contador = documentos únicos** ("1 fonte" / "N fontes"), não trechos.
- **Pílula menor** (fonte ~11px, padding vertical menor, ícone `size-3`).
- As **citações inline `[n]` não mudam** — continuam mapeando para o trecho exato (`sources[n-1]`); só a linha de baixo é enxugada.

### 1b. Cursor de streaming inline

O cursor (caret) passa a ficar **no mesmo fluxo do texto**, colado ao último caractere gerado (inline, não em bloco abaixo). Some no `done`. Não deve causar reflow/"pulo" perceptível a cada flush (~60ms).

**Arquivos:** `explore-view.tsx` (render do markdown em streaming + caret inline).

**Critério:** durante a geração, o caret aparece imediatamente após o texto; não há elemento solto abaixo.

### 1c. Copiar resposta

Botão **Copiar** na barra de ações da resposta (copia o texto final; feedback "copiado"). Ao lado, reservar o slot do **Baixar** (Fatia 2), desabilitado/oculto por ora.

**Arquivos:** `explore-view.tsx` (barra de ações abaixo da resposta concluída).

## Fatia 2 — Baixar resposta (PROJETO PARA DEPOIS)

Milestone: **M2 — Explorar: baixar resposta**.

- Exportar a resposta em **PDF / Markdown / DOCX**.
- Ao clicar → envia a uma **fila** que **processa, gera e devolve** o arquivo. **Não é persistido** (efêmero).
- Provável reuso do worker/render existente (o publish já gera PDF via Puppeteer).
- Fora de escopo agora: só registrar como projeto/issue.

**Emenda (2026-07-07, ao implementar) — design travado:**

- **Síncrono, não fila.** O conteúdo da resposta é client-side (vai no request de qualquer jeito), então a fila seria cerimônia. Novo endpoint autenticado `POST /explore/export` `{ format: 'pdf'|'md', title?, content }` gera na hora e **devolve os bytes** (efêmero de verdade — nada persistido).
  - **pdf**: `markdownToHtml(content)` → `PuppeteerPdfRenderer.render({ title, contentHtml })` (ambos já existem) → `application/pdf`.
  - **md**: passthrough do `content` → `text/markdown`.
  - Resposta com `Content-Disposition: attachment; filename="<slug>.{pdf,md}"`; `title` fallback `"Resposta"`, filename sanitizado.
- **Front:** o botão **Baixar** (slot reservado na Fatia 1) vira `DropdownMenu` (PDF / Markdown); escolha → `exportAnswer` (via `apiFetchBlob`) → `downloadBlob(blob, filename)` (`<a download>`). Loading no botão; erro com feedback breve.
- **DOCX fica para depois** (precisa de lib nova de backend) — fast-follow.
- Sem mudança no core, sem fila, sem MinIO.

## Fatia 3 — Busca semântica (ÉPICO PARA DEPOIS)

Milestone: **M3 — Busca semântica**.

- Nova feature "Busca": campo de busca → lista de **trechos relevantes** (do `/search`, que já existe no backend) → clicar leva ao **documento/arquivo no ponto**.
- Distinta do command-palette atual (Alt+B), que só filtra **títulos** de documentos client-side (não busca conteúdo).
- Decisões abertas (resolver no épico): tela dedicada vs. evoluir o Alt+B; realce/scroll até o trecho no doc.

---

## Fora de escopo / não fazer agora

- Mudança nos contratos de backend do `/explore` (o dado das fontes já basta).
- Rolar/realçar o trecho exato dentro do documento aberto (stretch da 1a).
- Persistir arquivos gerados na Fatia 2.

## Constraint do projeto

Sem qualquer atribuição de IA em commits/PRs/código (regra do dono do repo).
